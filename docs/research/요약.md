# 프로젝트 요약: Jupyter + LLM 커널 디버깅 분석 시스템

## 개요

**목표**: JupyterLab 환경에 LLM을 연결하여 Android/Linux 커널 메모리 덤프(ramdump) 및 vmlinux 파일을 AI가 자동 분석해주는 시스템 구축

**현황**: Pilot test 완료 (4GB Linux memory dump 분석 완료)

---

## 핵심 기능 (원하는 것)

| # | 기능 | 설명 |
|---|------|------|
| 1 | **대화형 장애 원인 분석** | 레지스터/콜스택을 붙여넣으면 LLM이 자연어로 해석 |
| 2 | **분석 스크립트 자동 생성** | "모든 CPU에서 PC가 이상한 것 찾아줘" → Python 코드 생성 |
| 3 | **메모리 비트 플립 탐지** | 하드웨어 결함 패턴 자동 감지 |
| 4 | **오프라인/에어갭 환경 지원** | ramdump가 외부로 유출되면 안 되므로 사내 LLM만 사용 |
| 5 | **팀 전체 공유** | 분석 노하우를 프롬프트 템플릿화해서 팀원과 공유 |

---

## 구현 방식 (4가지 옵션)

| 방식 | 내용 | 선택 이유 |
|------|------|----------|
| **A. Jupyter AI** | 공식 Project Jupyter 산하 확장, %%ai 매직 커맨드 지원 | OpenAI-compatible API가 있을 때 가장빠르게 도입 |
| **B. Notebook Intelligence (NBI)** | 다양한 LLM을 유연하게 교체 | 다양한 LLM 프로바이더를 전환하고 싶을 때 |
| **C. Python 직접 통합** | LLMAssistant 헬퍼 클래스 + IPython 매직 커맨드 | 사내 API 규격에 무관하게 바로 적용 가능 |
| **D. 커스텀 MCP 서버** | LLM 에이전트가 ramdump 분석 함수를 직접 호출 | 팀 전체 표준 도구로 발전 가능 |

## 기술 스택

```
┌─────────────────────────────────────────────┐
│              JupyterLab UI                  │
├─────────────────────────────────────────────┤
│   Jupyter AI / NBI / 커스텀 매직 커맨드      │
├─────────────────────────────────────────────┤
│        LLM (Ollama / 사내 API / vLLM)       │
├─────────────────────────────────────────────┤
│  분석 도구: PyKdump, GDB, RAMParser, pyELF  │
├─────────────────────────────────────────────┤
│      데이터: ramdump, vmlinux, 커널 로그     │
└─────────────────────────────────────────────┘
```

**핵심 라이브러리**
- `LangChain` — LLM 에이전트 + 도구 호출
- `MCP (Model Context Protocol)` — 에이전트 ↔ 분석 도구 통신
- `PyKdump` — 커널 덤프(vmcore) Python 바인딩
- `pyELFtools` — vmlinux ELF 심볼 파싱
- `Ollama` — 로컬 LLM 서빙

---

## LLM이 사용할 분석 도구 (MCP 서버 제공)

```python
get_registers(cpu_id)          # CPU 레지스터 조회
read_memory(address, size)      # 메모리 주소 값 읽기
lookup_symbol(address)          # 주소 → 커널 함수명 변환
get_callstack(cpu_id)           # 콜스택 추출
detect_bitflip(start, end)      # 비트 플립 검출
get_kernel_log(last_n_lines)    # dmesg 버퍼 추출
read_variable(variable_name)    # 커널 전역 변수 읽기
```

## 분석 3계층 구조

| 계층 | 역할 | 사용 모델 |
|------|------|----------|
| 1층 (반사) | 간단한 패턴 추출 (정규식) | Python 기본 |
| 2층 (빠른 사고) | 간단한 헥사 덤프 이상 판별 | Qwen2.5-Coder 7B |
| 3층 (전문가) | 복합 root cause 분석 | MiMo-V2 / DeepSeek-Coder 33B |

---

## 보안 요구사항 (필수)

- ✅ ramdump/vmlinux는 **절대 외부 전송 금지**
- ✅ LLM은 **사내 온프레미스 또는 Ollama 기반 로컬** 모델만 사용
- ✅ LLM에는 실제 메모리 원본이 아닌 **분석 결과 요약**만 전달
- ✅ API 키는 환경 변수 또는 Vault로 관리
- ✅ 오프라인(에어갭) 환경 설치 지원 (`wheelhouse` 기법)

| 단계 | 작업 | 예상 기간 |
|------|------|----------|
| 1 | 사내 LLM API 연결 테스트 | 1~2일 |
| 2 | `LLMAssistant` 헬퍼 클래스 개발 | 2~3일 |
| 3 | IPython 매직 커맨드(`%%ai`, `%%ai_analyze`) 개발 | 1~2일 |
| 4 | Jupyter AI 또는 NBI 설치/설정 | 1일 |
| 5 | 프롬프트 템플릿 라이브러리 구축 | 3~5일 |
| 6 | 팀 배포 및 피드백 수집 | 1주 |
| 7 | MCP 서버 개발 (선택) | 2~3주 |

## 기대 효과

1. 수작업 분석 시간 **수 시간 → 수 분** 단축
2. 신입/경험이 적은 팀원도 AI 가이드를 통해 독립적으로 분석 가능
3. 분석 노하우가 프롬프트 템플릿으로 축적되어 **팀 지식베이스** 형성
4. 반복적 분석 작업의 **완전 자동화** 가능
